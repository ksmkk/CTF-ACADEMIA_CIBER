XSS (Cross-Site Scripting) - Mini Desafio 2

DOM-Based XSS

Objetivo

Descubre cómo el código JavaScript del lado del cliente puede ser vulnerable a XSS. A diferencia del anterior, aquí el servidor NO participa en la inyección: el navegador lo hace.

Informacion: DOM-Based XSS

DOM-Based XSS ocurre cuando:

  • El código JavaScript lee datos del usuario (URL, input, etc.)
  • Los inyecta directamente en el DOM usando innerHTML, eval(), etc.
  • El navegador ejecuta el código como JavaScript
  • El servidor NUNCA ve el contenido malicioso (por eso es difícil de detectar)

Diferencia clave: En este XSS, el servidor es inocente. El JS del lado del cliente es quien comete el error de permitir la inyección.

Codigo Vulnerable

// ❌ VULNERABLE:
const name = document.getElementById('input').value;
document.getElementById('output').innerHTML = 'Hola ' + name;

¿Problema? Si name contiene <img src=x onerror=alert("XSS")>, se ejecutará.

// ✅ SEGURO:
const name = document.getElementById('input').value;
document.getElementById('output').textContent = 'Hola ' + name;

¿Ventaja? textContent NO interpreta HTML, solo muestra texto.

Comparativa: innerHTML vs textContent

Propiedad ¿Interpreta HTML/JS? ¿Segura?
innerHTML ✅ Sí (peligroso) ❌ No
textContent ❌ No ✅ Sí
eval() ✅ Sí (muy peligroso) ❌ Nunca
